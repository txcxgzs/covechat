# CoveChat 项目上下文与续作清单

> 最后更新：2026-07-19
> 仓库：<https://github.com/txcxgzs/covechat>  
> 当前状态：`0.1.0-experimental`，未经过独立安全审计，不得宣传为“绝对安全”或用于高风险敏感通信。

## 1. 我们在做什么

CoveChat 是面向国际市场、采用 AGPL-3.0 的开源端到端加密聊天项目。

首发目标：

- 可安装的网页 PWA，默认中文，完整支持中文和英文。
- Linux 服务端优先，提供 Docker Compose 一键部署。
- Web 服务默认只监听 `127.0.0.1`，方便通过 Nginx、Caddy 等反向代理提供 HTTPS。
- 支持用户名账户、恢复码、本地口令解锁。
- 支持端到端加密单聊、最多 50 个成员设备的小群、文本、图片和文件。
- 支持离线消息、设备列表、远程撤销、安全码验证、身份密钥变化阻断、加密云备份、定时消失、主动举报、拉黑及限流。
- 后续复用 Rust 协议核心开发 iOS/Android 客户端。

## 2. 安全目标和边界

目标：

- PostgreSQL、Redis、对象存储和应用服务器完全泄露时，攻击者仍不能读取消息正文、附件密钥、原始文件名、MIME、群消息或备份明文。
- 防御网络监听、消息篡改、重放、恶意预密钥、恶意管理员、账户劫持和短期设备密钥泄露。
- 提供前向保密、入侵后恢复、逐设备身份、安全码验证和密钥变化告警。
- 所有密码学操作放在 Rust 核心，通过 WASM 暴露最小接口；业务代码不得自行实现加密算法。

不承诺防御：

- 已被恶意软件、键盘记录器或恶意浏览器扩展完全控制的终端。
- 截图、拍照或收件人主动复制消息。
- 首版不提供洋葱路由，也不抵抗国家级流量关联分析。

发布规则：

- 独立安全审计完成前必须保持 `experimental/unaudited` 标识。
- 不得把社区审查描述成专业审计。
- 解密失败、未知协议版本或身份变化必须失败关闭，禁止降级为明文。

## 3. 技术架构

- 前端：React + TypeScript + Vite PWA。
- 密码学核心：Rust + WASM。
- 单聊：官方 Signal `libsignal` 固定提交，PQXDH、Kyber、Double/Triple Ratchet、逐消息密钥。
- 群聊：OpenMLS 0.8.1，RFC 9420 MLS，成员变化推进 epoch。
- API/WebSocket：Rust + Axum。
- 数据库：PostgreSQL + SQLx migrations。
- 分布式限流：Redis。
- 附件：S3 兼容对象存储（部署默认 MinIO）。
- 部署：Docker Compose；Linux 使用 `deploy.sh`，Windows 可使用 `deploy.ps1`。

## 4. 已完成内容

### 4.1 部署与工程

- 已建立 monorepo、AGPL-3.0、CI、Dockerfile、Compose、README 和安全文档。
- Linux `deploy.sh`：

  - `umask 077` 保护 `.env`。
  - 默认 `127.0.0.1:8088`。
  - 支持 `--host`、`--port`。
- `compose.deploy.yml` 只暴露 Web 入口；API、PostgreSQL、Redis、MinIO 不直接暴露公网。
- GitHub Actions 检查前端构建、Rust fmt、全特性 Clippy、全工作区测试和 WASM 编译。
- 截至提交 `04bf4c6` 的 CI 已全部通过。

### 4.2 账户、设备和恢复

- 浏览器生成账户签名密钥、设备签名密钥和 256 位带校验恢复码。
- 本地口令经 Argon2id 派生密钥，使用 XChaCha20-Poly1305 加密保险库。
- 私钥、Signal 状态、MLS 状态和信任状态加密后保存在 IndexedDB。
- 设备私钥签署服务端一次性挑战登录。
- 支持恢复码认证、加密云备份恢复和恢复后设备凭据轮换。
- 支持查看账户所有设备和远程撤销其他设备。
- 撤销设备不能继续登录或接收新投递。

### 4.3 单聊

- 已接入固定版本的官方 `libsignal`。
- 支持 PQXDH/Kyber 离线建链、Double/Triple Ratchet、乱序状态持久化。
- Ratchet 状态先持久化再发送密文，避免崩溃导致消息密钥复用。
- 一次性预密钥消费后自动轮换并重新发布。
- 已有测试验证持久化、篡改失败和身份密钥替换阻断。
- 已支持真实用户名收件人、WebSocket 通知和离线邮箱。
- 本地聊天历史已加密持久化，不再依赖演示消息。

### 4.4 身份验证

- 信任记录使用 Rust/WASM 独立的 HKDF/AAD 密钥域加密。
- 首次观察账户和设备身份后记录指纹。
- 同一账户签名密钥或同一设备身份指纹变化时立即阻断发送。
- 安全面板可显示基于双方账户/设备身份计算的真实安全码并标记已验证。

### 4.5 MLS 群聊

- 已接入 OpenMLS 0.8.1 / RFC 9420。
- 支持创建设备 KeyPackage、创建群、邀请、Welcome、加密群消息、处理 commit 和移除成员。
- 群最大限制为 50 个成员设备。
- 成员变化触发新 epoch；被移除成员不能解密新 epoch 消息。
- MLS 状态加密持久化，KeyPackage 使用后轮换并重新发布。
- 已在两个真实浏览器身份间完成创建群、邀请、Welcome、群消息解密和刷新恢复测试。

### 4.6 附件

- 客户端生成随机文件密钥并按 1 MiB 分块加密。
- 文件密钥、文件名、MIME 和缩略元数据只放入 Signal 端到端加密载荷。
- 服务端只保存随机对象 ID、密文分块、摘要、密文大小和到期时间。
- 下载时验证 manifest、每块摘要、对象上下文、块序号和最终明文大小。
- 已补齐“上传后通过 Signal 密文把附件引用发送给收件人”的闭环。

### 4.7 备份

- 服务端强制备份版本单调递增和 `previousDigest` 哈希链，拒绝回滚或错误父版本。
- 备份由恢复秘密经过域隔离派生的密钥在客户端加密。
- 新版备份载荷包含账户材料、Signal 状态、MLS 状态、信任状态和本地历史。
- 协议状态变化后会串行触发备份同步，避免并发写破坏版本链。
- 仍需做真实断网重试、冲突恢复和多版本迁移的完整 E2E 验收。

### 4.8 滥用控制

- 用户可明确选择最近收到的消息并在确认提示后主动披露给服务端举报。
- 举报请求由设备密钥签名，限制正文和上下文大小。
- 举报和拉黑关系持久化到 PostgreSQL。
- 拉黑在密文投递层生效；被拉黑用户无法向对方设备投递新消息或 MLS 邀请。
- Redis 已接入跨实例消息和举报限流；Redis 故障时生产限流路径失败关闭。
- Redis Pub/Sub 跨 API 实例邮箱事件总线已写入当前工作区：投递后本机立即通知，同时发布到 Redis；各 API 实例订阅统一频道并转发给本机 WebSocket。

### 4.9 双语和 PWA

- 默认语言为中文，可切换英文并持久化选择。
- PWA 可安装，使用生成式 Service Worker 预缓存自有静态资源和 WASM。
- 禁止第三方运行时脚本，配置了 CSP、Trusted Types 和安全响应头。
- 所有新增界面必须继续同时提供中文和英文。
